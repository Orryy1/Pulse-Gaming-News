# Render Fallback Root Cause Analysis & Fix

**Date:** 2026-04-30 evening
**Affected story:** `rss_4ff45649e69b89de` ("The next Tales Of remaster has leaked, and it's probably not what you're expecting") — published at 10:03 UTC.
**Symptom:** Multi-image ffmpeg filter graph crashed silently. Story rendered as `legacy_single_image_fallback · class=fallback · visuals=8` despite 8 images being available in the deck.
**Mission constraints honoured:**

- ❌ Did NOT change production render defaults.
- ❌ Did NOT switch Studio V2 into production.
- ❌ Did NOT enable hard blocking gates.
- ❌ Did NOT broaden scope into new creative features.

## 1. Investigation: looking for the exact ffmpeg error

Checked Railway logs first via `railway logs --service marvelous-curiosity` filtered for "Multi-image render FAILED", "Tales Of", `rss_4ff45649`, "complex filtergraph". **No matches** — Railway log retention had rolled past this morning's 10:03 UTC failure by the time investigation started.

Inspected `assemble.js` line-by-line and found the smoking gun in the existing code at lines 1834–1846 — a prior 2026-04-25 hotfix comment that documents this exact failure class:

> "The Sharp mozjpeg output produces JPEGs whose chroma metadata ffmpeg's auto-inserted swscaler can't reconcile when stitched alongside PNG overlays — every multi-image render fails with `[auto_scale_N] Failed to configure output pad on auto_scale_N` / `Failed to inject frame into filter network: Resource temporarily unavailable` and the produce cycle ends with 0 MP4s."

The April 25 hotfix mitigated by reverting the smart-crop integration. Since then, the multi-image deck has continued to mix sources of varying provenance:

- Steam capsule (JPEG, 600×900)
- Steam hero (JPEG, ~1920×620)
- Steam screenshots (JPEG, varied)
- IGDB cover (JPEG via `images.igdb.com`, varied)
- IGDB screenshots (JPEG, 1080p variant)
- Article scrape (mixed JPEG / PNG / WebP, varied)
- Article hero / og:image (mixed format / dimension)
- Pexels stock (JPEG, varied)
- Bing scrape (mixed format)
- Reddit thumbnail (JPEG/PNG/WebP)

Every one of these sources can produce a file that _passes_ the existing `>10KB` cache filter but contains:

- corrupt EOI markers (truncated download)
- 0×0 dimensions reported by libjpeg-turbo
- single-pixel decode after EXIF-orientation handling
- WebP that ffmpeg's libwebp build can't handle
- JPEGs with non-standard chroma subsampling that the swscaler rejects

When ANY one input in a deck of N fails to decode, the entire `xfade` chain fails because `[v(i+1)]` depends on `[v(i)]`. The catch block at line 1915 then drops to single-image fallback.

## 2. Identified cause class

**Most likely**: corrupt or unusual-format input image AND/OR chroma-format mismatch between mixed-source inputs.

Other classes considered and ruled less likely for THIS specific failure:

- Path escaping: Tales-Of story id is `rss_4ff45649e69b89de` — no special chars; paths use forward slashes and double-quote wrapping.
- Filter labels: filter graph is generated by a single function with deterministic labelling.
- Concat issue: the renderer uses `xfade`, not `concat`.
- Missing media: the image-cache `>10KB` filter and `realImages` accept-list filter would have caught a missing file.
- Drawtext escaping: today's earlier `sanitizeDrawtext` already strips ffmpeg-hostile chars.
- ASS escaping: `assPath` is sanitised at line 1300.

The Tales-Of failure is consistent with one of the 8 inputs failing to decode through ffmpeg's auto-inserted swscaler. The Discord publish at 20:04 UTC for "GTA 6 Owner Passed On A Sequel" rendered cleanly with the same code path and 5 visuals — so the failure is per-input, not systematic.

## 3. Fix shipped

**Smallest root-cause fix.** No render-default change. No new creative feature.

### `lib/render-input-validation.js` (new module)

Pure helpers, dependency-injected `sharp` for testability:

- `validateImageFile(path, opts)` — opens the file with sharp, checks:
  - file exists
  - file size ≥ 200 bytes
  - sharp can read metadata
  - width and height are finite, > 0
  - dimensions ≥ 32×32 (below this, ffmpeg's scale+crop fails)
    Returns `{ ok, width, height, reason }`.
- `validateImageBatch(paths)` — runs validation across an ordered list. Returns `{ good: [...], bad: [{ path, reason }] }`. `good` preserves input order so caller priority is intact.
- `classifyFfmpegError(stderr|err)` — matches stderr against a 16-class lookup table:
  - I/O classes: `input_no_such_file`, `input_invalid_data`, `input_decode_error`, `input_unsupported`
  - Filter classes: `filter_graph_parse_error`, `filter_label_not_found`, `filter_param_error`
  - Overlay classes: `drawtext_error`, `ass_error`, `ass_path_error`
  - Resource: `input_count_exceeded`, `memory_error`, `killed_by_signal`, `timeout`
  - Math: `xfade_offset_error`, `concat_error`
  - Catch-all: `ffmpeg_unknown`
- `buildFallbackReason({ errorClass, detail, inputsValidated, inputsBad })` — produces the structured `render_fallback_reason` string capped at 400 chars.

### `assemble.js` changes

Two narrow edits, no rewrite:

1. **Pre-flight validation block** inserted just after `const images = rawImages;` (line ~1846) and before `buildVideoCommand`. Validates every input path in the deck. Bad inputs are dropped with reason logged. If validation drains the deck completely, falls back to the composite image (still produces a video) and stamps `class=all_inputs_invalid`. If validation succeeds with a reduced deck, multi-image render proceeds with the survivors.

2. **Catch-block reason classification** upgraded. The existing forensic stamp (shipped in `c0f1707`) captured the raw stderr tail. The new code classifies stderr into one of the 16 enum classes via `classifyFfmpegError` and packages it via `buildFallbackReason`. Operator now sees:
   ```
   class=filter_graph_parse_error | inputs_validated=8 | inputs_bad=0 | detail=Error parsing filtergraph...
   ```
   instead of just the raw stderr.

The pre-flight is wrapped in try/catch so a module-load failure of `lib/render-input-validation.js` is non-fatal: the multi-image render still proceeds against the original list and the existing catch handles failures as before.

## 4. Tests added

**29 new unit tests** in `tests/services/render-input-validation.test.js`, all passing:

### `classifyFfmpegError` (14 tests)

- 'No such file' → `input_no_such_file`
- 'Invalid data' → `input_invalid_data`
- 'Could not decode' → `input_decode_error`
- Filter graph parse error
- Filter label not found
- ASS path error (the 2026-04-25 hotfix class)
- Drawtext failure
- SIGKILL / OOM / timeout
- Unrecognised input → `ffmpeg_unknown` (catch-all)
- null/undefined inputs
- Error object via `.stderr` and `.message`

### `validateImageFile` (7 tests)

- Missing path → `no_path`
- File missing → `stat_failed`
- Too-small file → `too_small`
- 0×0 dims → `zero_dimensions`
- Tiny dims (8×8) → `dim_too_small`
- Sharp throws → `sharp_decode`
- Healthy 1080×1920 image → `ok=true`

### `validateImageBatch` (2 tests)

- Mixed batch (good + missing + tiny) → drops bad, preserves order
- Empty input → empty arrays

### `buildFallbackReason` (4 tests)

- Includes class + counts + truncated detail
- Total length capped at 400
- Missing args → `class=ffmpeg_unknown`
- Omits `inputs_bad` when zero

### Registry sanity (2 tests)

- Every entry has `class` + `re`
- Classes are unique

## 5. Before / after behaviour

| Scenario                            | Before                                                | After                                                                             |
| ----------------------------------- | ----------------------------------------------------- | --------------------------------------------------------------------------------- |
| Healthy 8-image deck                | Multi-image render                                    | Multi-image render (no behaviour change)                                          |
| 1 corrupt image in deck of 8        | Whole graph crashes → single-image fallback           | Validator drops the 1 bad input → multi-image render with 7                       |
| All inputs corrupt                  | Single-image fallback (composite) with mystery reason | Single-image fallback (composite) with `class=all_inputs_invalid \| inputs_bad=N` |
| ffmpeg crashes despite valid inputs | `render_fallback_reason` = raw 400-char stderr        | `render_fallback_reason` = `class=<enum> \| inputs_validated=N \| detail=<tail>`  |
| Validation module load fails        | n/a (didn't exist)                                    | Try/catch falls through to original list — multi-image render still proceeds      |

**Production render default unchanged.** Multi-image is still the canonical path. Single-image fallback still fires when the multi-image graph fails. The only behavioural changes are:

1. Bad inputs are dropped before the graph is built (resilience).
2. The `render_fallback_reason` field carries a structured class enum (observability).
